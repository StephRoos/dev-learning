3.4 — Code Ninja (anti-patterns)

Qu'est-ce que le "code ninja" ?

Le "code ninja" est du code volontairement (ou involontairement) difficile à lire, à maintenir ou à comprendre. Il impressionne parfois à première vue, mais c'est un piège.

Règle d'or : Code is read far more often than it is written. Écris pour le prochain développeur (qui sera souvent toi-même dans 6 mois).

Les anti-patterns classiques

1. Noms d'une lettre : let a, b, c, d, i, j, x, y
Exception tolérée : variables de boucle simples (i, j) et coordonnées mathématiques (x, y).
2. Abréviations cryptiques : let usr, nbr, str, tmp, chk, rslts
Écris : let utilisateur, nombre, texte, temporaire, verifier, resultats
3. Réutilisation de variables : utiliser data pour stocker 3 choses différentes dans la même fonction.
4. Underscore préfixe mal utilisé : _nom implique "privé" — utilise-le seulement pour signaler qu'une propriété est interne.
5. Effets de bord cachés : une fonction nommée estValide() qui modifie aussi des données — surprise !
6. Opérateurs ternaires imbriqués : x ? y ? a : b : z ? c : d — illisible au-delà d'un niveau.
7. Typage implicite abusé : if (obj) pour tester 4 cas différents en même temps (null, undefined, 0, '').
